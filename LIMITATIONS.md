### Known resolve/typecheck-level divergences (tracked, deliberate тАФ tightened by mt-022/025/026/023; **the whole residual below is bead mt-027**, owner-accepted-and-deferred 2026-07-16: revisited as Rung-3/4 typing work deepens the same machinery)
All of these lean toward **over-acceptance** (mettle never wrongly rejects). The differential gauge ([docs/reference/alloy4fun-resolve-pass.md](docs/reference/alloy4fun-resolve-pass.md)) measures every one against the jar over the **150,891 unique alloy4fun codes** and the 167-file corpus. **mt-025** built the ADR-0008-decision-4 **materialized two-pass** structure for real (bottom-up typed tree with join-level `ExprChoice`s, faithful top-down `resolve(relevantType)` + `resolveHelper`, per-call `DeduceType` return specialization, `this/`-qualified own-module scope): over-acceptance fell **2,475 тЖТ 320** (98.36% тЖТ **99.79%** agreement), still **0 jar-accepts/mettle-rejects** and corpus 167/167. The `ambig`/`join_lenient`/loose-want suppression heuristics of mt-018/022 are **gone** тАФ suppression is now structural (a choice resolves only its chosen candidate; errors on discarded readings never surface). **mt-026** then closed the 6 of those 320 that were actually a parser gap, not a resolve gap (see "Resolved this session (mt-026)" below): **320 тЖТ 314**, still 99.79% (150,891 codes makes 6 codes too small to move the rounded percentage). Frequencies below are absolute counts over the 150,891 codes at the mt-025 state (pre-mt-026; the parser-level 6 are no longer part of this resolve-level bucket).
- **Left-of-join field ambiguities in a few deep contexts still lean accept (~94 codes; was ~1,281).** The dominant `projects.p` / `s.projects` case now **rejects faithfully** (the join-level choice resolves against the precise parent relevant type, `resolveHelper`'s `hasCommonArity` fallback keeping both fields when the slice is empty). The residual ~94 are joins whose left operand's type mettle only approximates (e.g. a function whose non-`univ` return it cannot fully specialize), so its join-level choice stays broader than the jar's.
- **An unknown name **inside a compound right operand** of a join is accepted (~61 codes: "cannot be found"; ~51 more: `*`/`^`/`~`-non-binary inside the same).** A bare unknown-name join RHS (`source.s`) now rejects; but `x . (compound)` тАФ e.g. `s.*next` тАФ keeps its bottom-up type without re-resolving the compound: resolving it standalone re-introduces the `next`/`integer/next` ambiguity the join is meant to settle (measured: 28,400 false rejects), so mettle leaves it. Catching these needs threading the join's *precise* right-slice into the compound, which the current fallback slice is not reliably precise enough for.
- **Failed function/predicate calls fall back to a relational join (~22 codes; was ~179).** The jar rejects "possible incorrect function/predicate call"; mettle's relational reading accepts an (often empty) join instead.
- **The multiplicity flag is not tracked (~25 "multiplicity not allowed", ~13 exactly-of).** An arrow/decl multiplicity used where a plain relation is required is not rejected (the reference's per-node `mult` flag; low frequency, deferred).
- **Some illegal joins remain (~21 codes; was 3,261) and a long tail (~10 "failed to be typechecked", misc).** `IllegalJoin` fires whenever a join's faithful type is truly empty (both operands unary), guarded off `univ`/`$` placeholders and compound-right operands.
- **Meta (`sig$`/`field$`) models resolve leniently**: `$`-bearing and unknown names in a `seen_dollar` model resolve to `univ` instead of running the reference's meta-sig synthesis phase (resolution-doc ┬з1 phase 8); all expression-level rejects are suppressed in such models.
- **The ┬з5.2 warning catalog is complete and measured to parity (mt-023).** All 20 warning classes are implemented under the reference's source-verified firing conditions; the warning *sets* match the jar on **166/167 corpus files (0 mettle-missing)** and **99.80% of the 101,970 agree-ACCEPT alloy4fun files** (98.6% warning recall). Warnings never affect the accept/reject verdict (in the reference or mettle тАФ LEDGER-002), so the conformance gauge is unaffected; `mettle check --strict` promotes them to a nonzero exit. Full methodology, stemтЖТclass table, and the honest small remainder (root-caused: `unused-var` inside overload-collapse-to-`none`; `eq`/`subset` "same value" on temporal var relations) are in [docs/reference/warning-parity.md](docs/reference/warning-parity.md).
- **Only `util/*` is embedded** as the last-resort module fallback; the reference jar serves *any* of its bundled `models/*` (book/example models) that way. A model opening a non-util jar-embedded module fails to load in mettle until a real model needs this. (The alloy4fun differential showed this is negligible: the only such cases were 6 codes whose jar-reject is a genuine parse error.)

### Rung-3 bounds builder (mt-030) тАФ deferred, never a wrong verdict
The bounds builder (`crates/als-core/src/bounds_builder.rs`, translation-ref ┬з1.4) is complete for the Rung-3 static slice; these pieces are intentionally deferred and cannot change a SAT/UNSAT verdict:
- **String is bound exactly empty.** mettle mints no string atoms yet (mt-029), so the `String` relation is `= {}`; full String support (referenced-literal collection, `maxstring` fill) is Rung 4 (ADR-0011).
- **The integer-ordering relations `Int/min`/`Int/max`/`Int/next`/`Int/zero` the jar's `A4Solution` also builds are not allocated.** They belong to Rung-4 integer fidelity (translation-ref ┬з9); Rung-3 models get a correct verdict without them. `Int` itself is bound exactly to the integer atoms and `seq/Int` to the first `maxseq` non-negatives, as the jar does.
- **`var` (temporal) sigs get static sibling disjointness** (`no (child_i & child_j)`), not the `[electrum]` temporal variant (translation-ref ┬з1.4). Temporal solving is Rung 6; a `var`-sig model does not yet solve, so the disjointness shape is moot until then.
- **`util/ordering` exact-bound pinning of `first`/`next` is not applied here** тАФ the ordering module's sigs/fields get ordinary bounds; the exact-constant shrink (LEDGER-004) is bead mt-035.

### Rung-3 IR lowering + goal assembly (mt-031) тАФ deferred, never a wrong verdict
The lowerer (`crates/als-core/src/lower.rs`, translation-ref ┬з2) turns a resolved command into the three-sorted-IR goal. It is correct for the Rung-3 relational slice; the following defer with a **typed** [`TranslateError`] (never a silent skip, never a wrong SAT/UNSAT verdict тАФ a deferred command simply does not solve yet):
- **Skolemization is skipped** (ADR-0011, translation-ref ┬з2.3): quantifiers are lowered directly (`some x: B | ╧Ж`), not turned into skolem constants. The verdict is identical; only the decoded instance differs тАФ a skolem relation the jar would show (`$cmd_x`) simply won't appear. Recorded here per the ADR.
- **The reference's ┬з2.5(4) reflexive `r = r` padding is not emitted.** It keeps unreferenced relations alive for Kodkod's solver; it is not a semantic constraint. mt-033 (CNF encoding) owns keeping bounded-but-unmentioned relations alive.
- **Temporal operators** (`always`/`until`/`'`/`var`) lower faithfully into the IR temporal kinds, but a command whose goal contains one returns `TranslateError::TemporalUnsupported` тАФ bounded LTLтЖТFOL solving is Rung 6.
- **String literals** in a command goal return `TranslateError::StringUnsupported` (Rung 4).
- **`pred/totalOrder[тАж]`** (the `util/ordering` `Ord` sig's appended fact, and explicit uses) returns `TranslateError::LoweringUnsupported` тАФ its exact-bounds/native-predicate handling is bead **mt-035** (LEDGER-004). Consequently most models that `open util/ordering` (or use an `enum`) defer this rung; the `first`/`next`/`prev`/`nexts`/`prevs` ordering **funcs** otherwise inline as ordinary funs.
- **Exotic field multiplicity shapes** defer: a **nested** multiplicity arrow in a field bound (`f: A -> (B -> one C)`) and `seq` field bounds return `TranslateError::LoweringUnsupported`. Flat unary bounds (`set`/`some`/`lone`/`one`/default-`one`) and a single binary arrow (`A m -> n B`) lower fully. The jar's redundant per-arrow-column membership constraints (`(v.f) in A`) are omitted тАФ they are entailed by the top-level `this.f in (A->B)`, so this is a sound simplification, not a gap.
- **Higher-order macros** (a macro receiving a callable passed by name, the resolver's accept-lean ┬з3.7 case) cannot be faithfully replayed and return `TranslateError::LoweringUnsupported`.
- **A `check` on a named pred, a `run` on a fun, and an overloaded `run`/`check` target** defer (uncommon shapes); named-pred `run` (params existentially quantified), block `run`/`check`, and named-assert `check` (negated) all lower.
- Corpus lowering coverage (all 167 files, every root command): **124 lower, 440 defer** (dominated by `pred/totalOrder`/`util/ordering` тЖТ mt-035, then temporal тЖТ Rung 6), **zero panics**, fully deterministic. Choice recording is additive and provably non-behavioral (the alloy4fun resolve gauge is byte-identical before/after over all 150,891 codes).

### Rung-3 CNF encoding + solve (mt-033) тАФ deferred, never a wrong verdict
The encoder + solve driver (`crates/als-core/src/encode/`, `src/solve.rs`, translation-ref ┬з4/┬з10.4) turn a lowered goal into CNF, solve with the mt-032 CDCL solver, and decode an instance. This is mettle's first real SAT/UNSAT verdict. The gauge is **verdict + SB-0 count** (ADR-0002); instance tuples are never diffed against the jar. Deferred pieces, each a **typed** `TranslateError` (never a wrong verdict):
- **Integer fidelity is the Rung-3 slice only.** A corpus measurement showed the only integer nodes reachable in lowerable commands are `Const` (36), `Card`/`#` cardinality (46), and `AtomToInt`/`int[┬╖]` (68) тАФ **zero** arithmetic, `sum`, or integer-`if/then/else`. Those three plus `IntToAtom`/`Int[┬╖]` and integer comparison are encoded (two's-complement, honoring the LEDGER-001 overflow switch). **Integer arithmetic (`plus`/`minus`/`mul`/`div`/`rem`/shifts), `sum`, and integer if-then-else return `TranslateError::LoweringUnsupported`** тАФ full integer/counting fidelity is Rung 4 (ADR-0011). A command using them defers rather than solving.
- **No symmetry breaking (by design).** Rung 3 ships zero lex-leader machinery, so mettle *is* the `symmetry = 0` regime and matches the jar's SB-0 count directly (ADR-0002); default-symmetry (SB=20) count parity is a later dedicated net.
- **SB-0 count parity holds only for goals without a skolemizable top-level existential.** mettle does not skolemize (ADR-0011, ┬з2.3), so its raw model count for a goal like `some x: A | ╧Ж` (or a `check` whose negation is a top-level `тИГ`) differs from the jar's, which counts the skolem constant relation's assignments too. The **verdict is always identical**; only the count differs. Pinned example (translation-ref ┬з10.4): `oracle/test1.als`'s `check NoEmpty` is SAT in both, jar SB-0 count 561 vs mettle 464. Goals without a top-level existential (e.g. `some r` = 1129) match exactly.
- ~~A pre-existing mt-029 scope-derivation bug~~ тАФ **FIXED at mt-033 review** (same session): the derivation rules now run as full passes per the reference (probe S1), so an abstract parent's unscoped children each inherit the parent scope and the difference rule can no longer back-derive a sibling to 0. The regression test (`abstract_unscoped_children_scope_bug`) is live (un-ignored); the 11 baseline disagreements it caused are gone.
- **Grounding is not env-cached this rung (perf, non-gating).** Quantifiers and comprehensions ground over their bound's candidate tuples, recomputing shared sub-expressions under each binding; nested quantifiers over larger scopes therefore produce large CNFs. Corpus coverage (all 167 files, 564 commands, post-scope-fix at the default 1s/command budget): **56 solved (28 SAT / 28 UNSAT), 440 lower-defer, 68 over-budget**, **zero panics**, deterministic (`METTLE_SOLVE_BUDGET_MS=5000` for the fuller sweep). Against the `baselines/` overlap: **one** disagreement remains тАФ `mediaAssets.als` `check PasteCut` (mettle SAT / jar UNSAT), a genuine encoder under-constraint for mt-034's self-check + mt-037 triage. Raising throughput (env-aware memoization, a `reduce_db` in the solver) is a later non-gating optimization (ADR-0011); the gauge is verdict + SB-0 count, not speed.

### Known syntax-level divergences from the reference (tracked, deliberate)
- **`steps`-scope validity checks deferred to resolve.** The jar rejects `for 1:2 steps` (increment must be 1) and unbounded steps not starting at 1 at command-build time; mettle currently parses these and will enforce the same checks when trace scopes are resolved (Rung 6). Never a wrong verdict тАФ commands don't solve yet.
- **Identifier Unicode classes approximate Java's** (`char::is_alphabetic`/`is_alphanumeric` vs `isJavaIdentifierStart/Part`); divergence is only possible for exotic non-ASCII identifiers. Now observed once: the mt-013 alloy4fun pass (186k records) found one submission using `тВм` (EURO SIGN, U+20AC) as an operand where `in` was meant тАФ `Character.isJavaIdentifierPart` classifies currency symbols as identifier characters, `char::is_alphanumeric` does not, so mettle lexes it as a stray character (a precise error) where the jar accepts it lexically and only rejects the resulting expression at type-check. mettle-rejects-earlier is not a wrong verdict on this file (both ultimately reject it), so left as-is; see [docs/reference/alloy4fun-error-pass.md](docs/reference/alloy4fun-error-pass.md) and [docs/reference/alloy6-grammar.md](docs/reference/alloy6-grammar.md) ┬з1.4.
- **Unterminated block comments** are a precise error in mettle; the reference's generated lexer silently ignores a `/*` that never closes. Impossible to hit on well-formed input.
- **Deep expression nesting is a precise `TooDeep` error in mettle, not a crash.** The reference jar suffers a raw `StackOverflowError` on pathological `(`/`{`/binder-chain nesting; mettle's parser now guards recursion depth (`MAX_EXPR_DEPTH = 256`, `crates/als-syntax/src/parser.rs`) and fails with a typed, spanned error instead. A deliberate, better-than-reference divergence (mt-014 Part 1); no real model approaches the limit. See [docs/reference/fuzzing.md](docs/reference/fuzzing.md) ┬з3.

### Resolved this session (mt-014)
- ~~**Binder-as-rightmost-operand (grammar-doc ┬з3.1) over-composes across nested precedence levels.**~~ **Resolved in mt-014.** The jar's actual rule was mapped empirically (тЙИ220 probes against the pinned jar) and implemented as an explicit "binder-composition budget" threaded through both the parser and the printer (`crate::prec::child_binder_budget`, shared so the two can never drift): a binder may be the rightmost operand of exactly one enclosing operator "hop"; a second hop is rejected unless the enclosing operator is a bare `implies` (no `else`) or the `else` branch of `implies тАж else`, which grant a fresh two-hop budget to their own branch; comparisons and the set-test prefixes (`no some lone one set seq`) never accept a binder operand at all. `q or r and all x: A | body` (the originally-reported shape) is now a precise `BinderNeedsParens` error ("a quantified formula here must be parenthesized"); all previously-accepted single-hop and `implies`-two-hop shapes are unaffected. Full probe table and rule derivation: [docs/reference/fuzzing.md](docs/reference/fuzzing.md) ┬з2.

### Resolved this session (mt-026)
- ~~**A prefix operator wrapping a binder falsely stayed transparent across an already-spent composition hop.**~~ **Resolved in mt-026.** mt-014's rule assumed every ordinary prefix (`!`/`not`, the temporal unaries, `# int sum`, `~ ^ *`) passes its ambient binder-composition budget through unchanged at any budget тАФ never actually verified for a prefix wrapping the binder itself (as opposed to an infix operator's *left* operand). It's wrong: 427 fresh jar probes showed a prefix wrapping a binder is transparent only while the ambient budget is still a fresh `TOP` (chains arbitrarily deep: `always always always all x: A | тАж` parses); once an ordinary infix operator has already spent its one hop, the prefix does not forward it тАФ `q and always all x: A | тАж`, `q and not some x: A | тАж` are syntax errors even though the un-prefixed `q and all x: A | тАж` is fine. Exactly 6 real alloy4fun codes hit this (`тАж and always all f: тАж | тАж` / `тАж and not some x: тАж | тАж` / `тАж iff not some x: тАж | тАж` shapes). Fixed with one new shared function, `crate::prec::prefix_operand_budget`, used by both the parser (`parse_prefix`, `parse_closure_at_depth`) and the printer (`write_unary`, `write_closure`) тАФ so the two stay in lockstep. Over-acceptance: 320 тЖТ **314**. Full probe table and rule derivation: [docs/reference/fuzzing.md](docs/reference/fuzzing.md) ┬з2 ("mt-026" subsection).
