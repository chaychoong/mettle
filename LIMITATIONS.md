# Limitations

**Status:** living document · honest and current. This file shrinks as rungs are completed. It never silently lies: anything mettle cannot yet do exactly is listed here, and unsupported constructs fail loudly ("parsed, not yet solvable"), never wrongly.

## Right now (Rungs 1–2 complete)
mettle lexes, parses, pretty-prints (`mettle parse`), and name/type-checks (`mettle check`) `.als` files — whole module worlds with `open`s, the clean-room stdlib, and rustc-style caret diagnostics into whichever file the error lives in. Gauge: 167/167 corpus accept; 150,891-code alloy4fun differential with **zero** jar-accepts/mettle-rejects and **99.79%** total agreement (over-acceptance 314, all documented below). Solving/visualization are later rungs; see [docs/ROADMAP.md](docs/ROADMAP.md).

### Known resolve/typecheck-level divergences (tracked, deliberate — tightened by mt-022/025/026/023; **the whole residual below is bead mt-027**, owner-accepted-and-deferred 2026-07-16: revisited as Rung-3/4 typing work deepens the same machinery)
All of these lean toward **over-acceptance** (mettle never wrongly rejects). The differential gauge ([docs/reference/alloy4fun-resolve-pass.md](docs/reference/alloy4fun-resolve-pass.md)) measures every one against the jar over the **150,891 unique alloy4fun codes** and the 167-file corpus. **mt-025** built the ADR-0008-decision-4 **materialized two-pass** structure for real (bottom-up typed tree with join-level `ExprChoice`s, faithful top-down `resolve(relevantType)` + `resolveHelper`, per-call `DeduceType` return specialization, `this/`-qualified own-module scope): over-acceptance fell **2,475 → 320** (98.36% → **99.79%** agreement), still **0 jar-accepts/mettle-rejects** and corpus 167/167. The `ambig`/`join_lenient`/loose-want suppression heuristics of mt-018/022 are **gone** — suppression is now structural (a choice resolves only its chosen candidate; errors on discarded readings never surface). **mt-026** then closed the 6 of those 320 that were actually a parser gap, not a resolve gap (see "Resolved this session (mt-026)" below): **320 → 314**, still 99.79% (150,891 codes makes 6 codes too small to move the rounded percentage). Frequencies below are absolute counts over the 150,891 codes at the mt-025 state (pre-mt-026; the parser-level 6 are no longer part of this resolve-level bucket).
- **Left-of-join field ambiguities in a few deep contexts still lean accept (~94 codes; was ~1,281).** The dominant `projects.p` / `s.projects` case now **rejects faithfully** (the join-level choice resolves against the precise parent relevant type, `resolveHelper`'s `hasCommonArity` fallback keeping both fields when the slice is empty). The residual ~94 are joins whose left operand's type mettle only approximates (e.g. a function whose non-`univ` return it cannot fully specialize), so its join-level choice stays broader than the jar's.
- **An unknown name **inside a compound right operand** of a join is accepted (~61 codes: "cannot be found"; ~51 more: `*`/`^`/`~`-non-binary inside the same).** A bare unknown-name join RHS (`source.s`) now rejects; but `x . (compound)` — e.g. `s.*next` — keeps its bottom-up type without re-resolving the compound: resolving it standalone re-introduces the `next`/`integer/next` ambiguity the join is meant to settle (measured: 28,400 false rejects), so mettle leaves it. Catching these needs threading the join's *precise* right-slice into the compound, which the current fallback slice is not reliably precise enough for.
- **Failed function/predicate calls fall back to a relational join (~22 codes; was ~179).** The jar rejects "possible incorrect function/predicate call"; mettle's relational reading accepts an (often empty) join instead.
- **The multiplicity flag is not tracked (~25 "multiplicity not allowed", ~13 exactly-of).** An arrow/decl multiplicity used where a plain relation is required is not rejected (the reference's per-node `mult` flag; low frequency, deferred).
- **Some illegal joins remain (~21 codes; was 3,261) and a long tail (~10 "failed to be typechecked", misc).** `IllegalJoin` fires whenever a join's faithful type is truly empty (both operands unary), guarded off `univ`/`$` placeholders and compound-right operands.
- **Meta (`sig$`/`field$`) models resolve leniently**: `$`-bearing and unknown names in a `seen_dollar` model resolve to `univ` instead of running the reference's meta-sig synthesis phase (resolution-doc §1 phase 8); all expression-level rejects are suppressed in such models.
- **The §5.2 warning catalog is complete and measured to parity (mt-023).** All 20 warning classes are implemented under the reference's source-verified firing conditions; the warning *sets* match the jar on **166/167 corpus files (0 mettle-missing)** and **99.80% of the 101,970 agree-ACCEPT alloy4fun files** (98.6% warning recall). Warnings never affect the accept/reject verdict (in the reference or mettle — LEDGER-002), so the conformance gauge is unaffected; `mettle check --strict` promotes them to a nonzero exit. Full methodology, stem→class table, and the honest small remainder (root-caused: `unused-var` inside overload-collapse-to-`none`; `eq`/`subset` "same value" on temporal var relations) are in [docs/reference/warning-parity.md](docs/reference/warning-parity.md).
- **Only `util/*` is embedded** as the last-resort module fallback; the reference jar serves *any* of its bundled `models/*` (book/example models) that way. A model opening a non-util jar-embedded module fails to load in mettle until a real model needs this. (The alloy4fun differential showed this is negligible: the only such cases were 6 codes whose jar-reject is a genuine parse error.)

### Rung-3 bounds builder (mt-030) — deferred, never a wrong verdict
The bounds builder (`crates/als-core/src/bounds_builder.rs`, translation-ref §1.4) is complete for the Rung-3 static slice; these pieces are intentionally deferred and cannot change a SAT/UNSAT verdict:
- **String is bound exactly empty.** mettle mints no string atoms yet (mt-029), so the `String` relation is `= {}`; full String support (referenced-literal collection, `maxstring` fill) is Rung 4 (ADR-0011).
- **The integer-ordering relations `Int/min`/`Int/max`/`Int/next`/`Int/zero` the jar's `A4Solution` also builds are not allocated.** They belong to Rung-4 integer fidelity (translation-ref §9); Rung-3 models get a correct verdict without them. `Int` itself is bound exactly to the integer atoms and `seq/Int` to the first `maxseq` non-negatives, as the jar does.
- **`var` (temporal) sigs get static sibling disjointness** (`no (child_i & child_j)`), not the `[electrum]` temporal variant (translation-ref §1.4). Temporal solving is Rung 6; a `var`-sig model does not yet solve, so the disjointness shape is moot until then.
- **`util/ordering` exact-bound pinning of `first`/`next` is not applied here** — the ordering module's sigs/fields get ordinary bounds; the exact-constant shrink (LEDGER-004) is bead mt-035.

### Rung-3 IR lowering + goal assembly (mt-031) — deferred, never a wrong verdict
The lowerer (`crates/als-core/src/lower.rs`, translation-ref §2) turns a resolved command into the three-sorted-IR goal. It is correct for the Rung-3 relational slice; the following defer with a **typed** [`TranslateError`] (never a silent skip, never a wrong SAT/UNSAT verdict — a deferred command simply does not solve yet):
- **Skolemization is skipped** (ADR-0011, translation-ref §2.3): quantifiers are lowered directly (`some x: B | φ`), not turned into skolem constants. The verdict is identical; only the decoded instance differs — a skolem relation the jar would show (`$cmd_x`) simply won't appear. Recorded here per the ADR.
- **The reference's §2.5(4) reflexive `r = r` padding is not emitted.** It keeps unreferenced relations alive for Kodkod's solver; it is not a semantic constraint. mt-033 (CNF encoding) owns keeping bounded-but-unmentioned relations alive.
- **Temporal operators** (`always`/`until`/`'`/`var`) lower faithfully into the IR temporal kinds, but a command whose goal contains one returns `TranslateError::TemporalUnsupported` — bounded LTL→FOL solving is Rung 6.
- **String literals** in a command goal return `TranslateError::StringUnsupported` (Rung 4).
- **`pred/totalOrder[…]`** (the `util/ordering` `Ord` sig's appended fact, and explicit uses) returns `TranslateError::LoweringUnsupported` — its exact-bounds/native-predicate handling is bead **mt-035** (LEDGER-004). Consequently most models that `open util/ordering` (or use an `enum`) defer this rung; the `first`/`next`/`prev`/`nexts`/`prevs` ordering **funcs** otherwise inline as ordinary funs.
- **Exotic field multiplicity shapes** defer: a **nested** multiplicity arrow in a field bound (`f: A -> (B -> one C)`) and `seq` field bounds return `TranslateError::LoweringUnsupported`. Flat unary bounds (`set`/`some`/`lone`/`one`/default-`one`) and a single binary arrow (`A m -> n B`) lower fully. The jar's redundant per-arrow-column membership constraints (`(v.f) in A`) are omitted — they are entailed by the top-level `this.f in (A->B)`, so this is a sound simplification, not a gap.
- **Field-group `disj` is dropped — a genuine under-constraint (mt-038 owns the fix, gate before mt-037; root-caused by mt-034).** A `disj a, b: … ` field group (and a post-colon `disj` bound) declares the named fields **pairwise disjoint** (`all this: S | no (this.a & this.b)`). The `Decl` AST carries `is_disj`/`is_bound_disj` (`als-syntax`), but `als_types::ResolvedField` does **not** record it, so the lowerer never synthesizes the disjointness — mettle's goal is strictly **weaker** than the jar's. This is the sole cause of the one corpus baseline disagreement, `mediaAssets.als[3]` `check PasteNotAffectHidden` (mettle finds a spurious counterexample where paste changes `hidden`, because `disj hidden, showing` is not enforced; jar UNSAT). The fix needs an `als-types` change (record the field-group disjointness) plus a lowering conjunct, so it is **not contained to `lower.rs`** — deferred to mt-038 (the pre-mt-037 lowering-gaps bead). Because the found instance genuinely satisfies mettle's own (too-weak) goal, the mt-034 self-check correctly **passes** on it: this is a lowering/synthesized-fact gap, not an encoder bug. Pinned as a known-limitation regression (`field_disj_dropped_known_gap` in `tests/solve.rs`).
- **Higher-order macros** (a macro receiving a callable passed by name, the resolver's accept-lean §3.7 case) cannot be faithfully replayed and return `TranslateError::LoweringUnsupported`.
- **A `check` on a named pred, a `run` on a fun, and an overloaded `run`/`check` target** defer (uncommon shapes); named-pred `run` (params existentially quantified), block `run`/`check`, and named-assert `check` (negated) all lower.
- Corpus lowering coverage (all 167 files, every root command): **124 lower, 440 defer** (dominated by `pred/totalOrder`/`util/ordering` → mt-035, then temporal → Rung 6), **zero panics**, fully deterministic. Choice recording is additive and provably non-behavioral (the alloy4fun resolve gauge is byte-identical before/after over all 150,891 codes).

### Rung-3 CNF encoding + solve (mt-033) — deferred, never a wrong verdict
The encoder + solve driver (`crates/als-core/src/encode/`, `src/solve.rs`, translation-ref §4/§10.4) turn a lowered goal into CNF, solve with the mt-032 CDCL solver, and decode an instance. This is mettle's first real SAT/UNSAT verdict. The gauge is **verdict + SB-0 count** (ADR-0002); instance tuples are never diffed against the jar. Deferred pieces, each a **typed** `TranslateError` (never a wrong verdict):
- **Integer fidelity is the Rung-3 slice only.** A corpus measurement showed the only integer nodes reachable in lowerable commands are `Const` (36), `Card`/`#` cardinality (46), and `AtomToInt`/`int[·]` (68) — **zero** arithmetic, `sum`, or integer-`if/then/else`. Those three plus `IntToAtom`/`Int[·]` and integer comparison are encoded (two's-complement, honoring the LEDGER-001 overflow switch). **Integer arithmetic (`plus`/`minus`/`mul`/`div`/`rem`/shifts), `sum`, and integer if-then-else return `TranslateError::LoweringUnsupported`** — full integer/counting fidelity is Rung 4 (ADR-0011). A command using them defers rather than solving.
- **No symmetry breaking (by design).** Rung 3 ships zero lex-leader machinery, so mettle *is* the `symmetry = 0` regime and matches the jar's SB-0 count directly (ADR-0002); default-symmetry (SB=20) count parity is a later dedicated net.
- **SB-0 count parity holds only for goals without a skolemizable top-level existential.** mettle does not skolemize (ADR-0011, §2.3), so its raw model count for a goal like `some x: A | φ` (or a `check` whose negation is a top-level `∃`) differs from the jar's, which counts the skolem constant relation's assignments too. The **verdict is always identical**; only the count differs. Pinned example (translation-ref §10.4): `oracle/test1.als`'s `check NoEmpty` is SAT in both, jar SB-0 count 561 vs mettle 464. Goals without a top-level existential (e.g. `some r` = 1129) match exactly.
- ~~A pre-existing mt-029 scope-derivation bug~~ — **FIXED at mt-033 review** (same session): the derivation rules now run as full passes per the reference (probe S1), so an abstract parent's unscoped children each inherit the parent scope and the difference rule can no longer back-derive a sibling to 0. The regression test (`abstract_unscoped_children_scope_bug`) is live (un-ignored); the 11 baseline disagreements it caused are gone.
- **Grounding is not env-cached this rung (perf, non-gating).** Quantifiers and comprehensions ground over their bound's candidate tuples, recomputing shared sub-expressions under each binding; nested quantifiers over larger scopes therefore produce large CNFs. Corpus coverage (all 167 files, 564 commands, post-scope-fix at the default 1s/command budget): **56 solved (28 SAT / 28 UNSAT), 440 lower-defer, 68 over-budget**, **zero panics**, deterministic (`METTLE_SOLVE_BUDGET_MS=5000` for the fuller sweep). Against the `baselines/` overlap: **one** disagreement remains — `mediaAssets.als[3]` `check PasteNotAffectHidden` (mettle SAT / jar UNSAT); mt-034's self-check **root-caused** it as an *under-constrained goal* (a dropped field-group `disj`, see the field-`disj` bullet below and §10.4), **not** an encoder bug — the mislabel "`check PasteCut`" in earlier notes is corrected (PasteCut is `[2]` and agrees, SAT/SAT). mt-038 owns the fix (mt-037 re-gauges). Raising throughput (env-aware memoization, a `reduce_db` in the solver) is a later non-gating optimization (ADR-0011); the gauge is verdict + SB-0 count, not speed.

### Rung-3 evaluator + self-check net (mt-034) — the "self-verified" promise
The direct three-sorted evaluator (`crates/als-core/src/eval.rs`, translation-ref §6) is an **independent second implementation** of the same semantics the encoder emits as SAT gates; it re-evaluates every found SAT instance against its own goal (`solve_goal`/`enumerate` `debug_assert!`; `als_core::self_check` in checked mode for tests/corpus). Two nets pass clean: the **encoder↔evaluator differential** (brute-force every candidate instance, assert the evaluator's accept count equals the solver's SB-0 count — 15 models spanning quantifiers, closure, subset sigs, field multiplicities, cardinality/int-compare, override, restriction, transpose, comprehension), and the **corpus self-check** (every solved SAT instance across all 167 files self-checks — **0 failures**). Scope-honest notes:
- **Integer arithmetic / `sum` / integer-`ITE` defer**, exactly as the encoder does — the evaluator handles only the Rung-3 integer slice (`Const`, `#`, `int[·]`, `Int[·]`, comparison), keeping it a *matched pair* with the encoder so the differential compares like for like. Overflow follows the LEDGER-001 switch: forbid (default) excludes an instance whose `#e` count or `int[·]` partial-sum leaves the signed range; allow wraps two's complement (translation-ref §2.4). A solver-produced instance never overflows, so the self-check never rejects one on overflow grounds — the path exists only to make the brute-force differential agree.
- **Temporal kinds return a typed error** (never reached: lowering defers temporal before solving).

### Known syntax-level divergences from the reference (tracked, deliberate)
- **`steps`-scope validity checks deferred to resolve.** The jar rejects `for 1:2 steps` (increment must be 1) and unbounded steps not starting at 1 at command-build time; mettle currently parses these and will enforce the same checks when trace scopes are resolved (Rung 6). Never a wrong verdict — commands don't solve yet.
- **Identifier Unicode classes approximate Java's** (`char::is_alphabetic`/`is_alphanumeric` vs `isJavaIdentifierStart/Part`); divergence is only possible for exotic non-ASCII identifiers. Now observed once: the mt-013 alloy4fun pass (186k records) found one submission using `€` (EURO SIGN, U+20AC) as an operand where `in` was meant — `Character.isJavaIdentifierPart` classifies currency symbols as identifier characters, `char::is_alphanumeric` does not, so mettle lexes it as a stray character (a precise error) where the jar accepts it lexically and only rejects the resulting expression at type-check. mettle-rejects-earlier is not a wrong verdict on this file (both ultimately reject it), so left as-is; see [docs/reference/alloy4fun-error-pass.md](docs/reference/alloy4fun-error-pass.md) and [docs/reference/alloy6-grammar.md](docs/reference/alloy6-grammar.md) §1.4.
- **Unterminated block comments** are a precise error in mettle; the reference's generated lexer silently ignores a `/*` that never closes. Impossible to hit on well-formed input.
- **Deep expression nesting is a precise `TooDeep` error in mettle, not a crash.** The reference jar suffers a raw `StackOverflowError` on pathological `(`/`{`/binder-chain nesting; mettle's parser now guards recursion depth (`MAX_EXPR_DEPTH = 256`, `crates/als-syntax/src/parser.rs`) and fails with a typed, spanned error instead. A deliberate, better-than-reference divergence (mt-014 Part 1); no real model approaches the limit. See [docs/reference/fuzzing.md](docs/reference/fuzzing.md) §3.

### Resolved this session (mt-014)
- ~~**Binder-as-rightmost-operand (grammar-doc §3.1) over-composes across nested precedence levels.**~~ **Resolved in mt-014.** The jar's actual rule was mapped empirically (≈220 probes against the pinned jar) and implemented as an explicit "binder-composition budget" threaded through both the parser and the printer (`crate::prec::child_binder_budget`, shared so the two can never drift): a binder may be the rightmost operand of exactly one enclosing operator "hop"; a second hop is rejected unless the enclosing operator is a bare `implies` (no `else`) or the `else` branch of `implies … else`, which grant a fresh two-hop budget to their own branch; comparisons and the set-test prefixes (`no some lone one set seq`) never accept a binder operand at all. `q or r and all x: A | body` (the originally-reported shape) is now a precise `BinderNeedsParens` error ("a quantified formula here must be parenthesized"); all previously-accepted single-hop and `implies`-two-hop shapes are unaffected. Full probe table and rule derivation: [docs/reference/fuzzing.md](docs/reference/fuzzing.md) §2.

### Resolved this session (mt-026)
- ~~**A prefix operator wrapping a binder falsely stayed transparent across an already-spent composition hop.**~~ **Resolved in mt-026.** mt-014's rule assumed every ordinary prefix (`!`/`not`, the temporal unaries, `# int sum`, `~ ^ *`) passes its ambient binder-composition budget through unchanged at any budget — never actually verified for a prefix wrapping the binder itself (as opposed to an infix operator's *left* operand). It's wrong: 427 fresh jar probes showed a prefix wrapping a binder is transparent only while the ambient budget is still a fresh `TOP` (chains arbitrarily deep: `always always always all x: A | …` parses); once an ordinary infix operator has already spent its one hop, the prefix does not forward it — `q and always all x: A | …`, `q and not some x: A | …` are syntax errors even though the un-prefixed `q and all x: A | …` is fine. Exactly 6 real alloy4fun codes hit this (`… and always all f: … | …` / `… and not some x: … | …` / `… iff not some x: … | …` shapes). Fixed with one new shared function, `crate::prec::prefix_operand_budget`, used by both the parser (`parse_prefix`, `parse_closure_at_depth`) and the printer (`write_unary`, `write_closure`) — so the two stay in lockstep. Over-acceptance: 320 → **314**. Full probe table and rule derivation: [docs/reference/fuzzing.md](docs/reference/fuzzing.md) §2 ("mt-026" subsection).

## How this file will be maintained
- As each rung lands, its capability moves out of "limitations" and the conformance scorecard records the exact agreement level.
- Constructs that parse but aren't yet solvable are listed explicitly and fail with a precise "not yet supported" diagnostic — never a wrong answer.
- Known permanent v1 non-goals (per plan §1): no native GUI (Sterling + CLI only), no unbounded model checking in v1 (temporal is bounded first), no obscure/rarely-used syntax corners until tracked here.
