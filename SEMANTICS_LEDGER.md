# Semantics Ledger

**Status:** living document · **Owner: human (product owner).** Agents implement *from* this ledger; they do not author settled rules into it.

## Purpose
For every corner where the reference Alloy jar's behavior *is* the spec, this ledger records a **one-sentence behavioral rule** plus a link to the conformance test that pins it. This is the understanding the project exists to produce. The rule is: read the reference until the behavior can be stated in one sentence, verify it empirically against the pinned jar, write it here with a test, then implement idiomatically.

## Entry format
```
### LEDGER-NNN — <corner>
Rule: <one sentence, testable>.
Status: proposed | verified | approved
Evidence: <how it was checked against the pinned jar>
Test: <path to the conformance test>
```
- `proposed` = drafted by an agent/tech lead, **not yet human-approved**.
- `verified` = confirmed empirically against the pinned jar, **with no open residual uncertainties** — every soft spot the drafting work flagged has been probed shut (or has amended the rule).
- `approved` = product owner has blessed it as canonical. **Only `approved` rules are safe to implement against.**
- Process rule (owner, 2026-07-16): an entry is put to the owner for approval **only at `verified`** — never with "re-derive later" caveats attached. If stating the rule wrong is a risk, the answer is more probes, not a caveated sign-off.
- Process rule (owner, 2026-07-18 — "Option B"): **no-fork entries** — entries that merely restate verified jar behavior with no genuine decision fork — are approved by the **tech lead** once `verified` (recorded as `approved (tech lead, no-fork delegation)`). Owner approval is reserved for entries containing a **genuine fork**: jar ambiguity/inconsistency (LEDGER-001's class), a deliberate divergence from the jar, or a resource/scope tradeoff. The tech lead must flag any fork explicitly and bring it to the owner framed as a decision with consequences — never buried in a status report.

### LEDGER-001 — integer overflow default ("forbid overflows" / `noOverflow`)
**Rule:** mettle treats integer/cardinality arithmetic exceeding the bitwidth as **forbidden by default** — an overflowing term excludes the instance (matching the Alloy GUI's default "Prevent overflows" = on). A `--[no-]overflow` flag switches to allow/wraparound semantics.
**Status:** `approved` (product owner, 2026-07-15). Facts below `verified`; safe to implement against.
**Evidence:** Alloy 6.2.0's default is **entry-point-dependent** (verified 2026-07-15, see [reference/alloy6-reference.md](docs/reference/alloy6-reference.md) §3(c) — reproduced by tech lead): GUI default = forbid (`noOverflow=true`); headless jar / `A4Options` default = allow/wraparound (`noOverflow=false`). Decisive test: `run { plus[7,7] < 0 } for 4 int` → **SAT** (allow) by default, **UNSAT** under `-n` (forbid). Default bitwidth = 4 (range −8..7).
**Decision (approved 2026-07-15):** canonical default = **forbid overflow**, to match the Alloy GUI's default experience (the Alloy users actually run). The conformance harness sets the oracle's `noOverflow` to match mettle's active setting so the scorecard always compares like-for-like.
**Test:** _(added with the Rung-3 integer work)_

### LEDGER-004 — `util/ordering` exact bounds & order pinning
**Rule:** Opening `util/ordering[S]` always forces sig `S`'s **total** population to be **exact** at whatever scope `S` resolves to — independent of the `exactly` keyword, of `S`'s multiplicity qualifier, and of whether `first`/`next`/`last`/`prev` are ever referenced. **(a)** When `S` has no proper subsig, or its subsigs leave no partition choice, the order's `first`/`last`/`next` are *additionally* bound to exact constants over `S`'s atoms in universe order (`first = S$0`, `last = S$<n-1>`, `next` = the consecutive-atom successor) — the linear order is fully pinned, independent of the symmetry-breaking setting. **(b)** When `S` has a proper subsig with non-exact scope, or two-or-more subsigs (even if individually exact), the constant binding does **not** engage: `first`/`next`/`last` are governed only by the ordinary `pred/totalOrder` constraint, and genuine order freedom (which chain rank carries which subsig tag) survives as multiple distinct instances at every symmetry setting.
**Status:** `approved` (product owner, 2026-07-16, on the amended two-part rule). Verified 2026-07-16: the exhaustive T10–T19 matrix closed every §9 residual and amended the rule (the original statement was correct for childless sigs but missed the subsig-conditional half; history below). Safe to implement against — unblocks bead mt-035.
**Evidence:** probes T4/T4b plus the mt-028 exhaustive matrix T10-T19 in [reference/alloy6-translation.md](docs/reference/alloy6-translation.md) §5/§10.1 (jar-verified 2026-07-16, OpenJDK 21, `oracle/org.alloytools.alloy.dist.jar` 6.2.0). Confirms: instance count = 1 at both symmetry 20 and symmetry 0 for a childless ordered sig at every tested size 2-6 (T10a-e), `next` always the plain consecutive chain `S$0->S$1->...->S$<n-1>` (resolves the old ">3-atom orders" residual); two independent ordered sigs pin independently (T12); an enum auto-opens ordering with the same pinning, `first` = the first **declared** constant (T13); the pinning is triggered purely by the `open` even when `first`/`next`/`last` are never referenced in the command (T19); a conflicting fact on `first` is UNSAT, proving the binding is a genuine constant not a solver preference (T16); a `var` ordered sig is rejected at parse/resolve with an explicit message (T18).
**IMPORTANT — the matrix also found a genuine counterexample to the rule as stated** (T14a-e, T15; resolves the old "partially scoped ordered sigs" residual, §9): when the ordered sig `S` has a proper subsig with **non-exact** scope, or **two-or-more** subsigs (even if each is individually `exactly`-scoped), the exact-constant shrink on `first`/`next`/`last` does **not** engage — `pred/totalOrder` is instead solved as an ordinary constraint, and genuine order freedom (which chain rank carries which subsig tag) survives as multiple distinct SAT instances at **both** symmetry 20 and symmetry 0 (e.g. T14b: 3 instances at sym20, 6 at sym0, all sharing the identical atom-name population `{A$0,A$1,B$0}` but with `B$0` at a different chain rank in each). The sig's **overall exact-scope forcing** (part (a) of the rule) is unaffected and holds unconditionally in every subsig configuration tested; only the **exact-constant** binding of first/next/last (part (b)) is subsig-conditional. An unrelated field reference to `S` (not a subsig) does not trigger this — T15 is a clean isolating control.
**Rule history:** the original mt-028 draft stated only the childless-sig behavior ("forces exact scope + binds first/last/next to exact constants, unconditionally"); the T14/T15 probes showed the constant-binding half is subsig-conditional, and the rule above is the amended statement (2026-07-16).
**Test:** _(added with the Rung-3 ordering work, mt-035 — must cover both the plain-sig exact-shrink path AND the subsig fallback path as two distinct behaviors, not one)_

### LEDGER-005 — integer wraparound & bitwidth (arithmetic at bitwidth)
**Rule:** At bitwidth `w` (default 4, signed range `−2^{w-1}..2^{w-1}−1`), the `fun/…` operators (`plus`/`minus`/`mul`/`div`/`rem`, shifts `<<`/`>>`/`>>>`, unary negate, `sum x|ie`, integer if-then-else) evaluate in `w`-bit two's-complement — **`div` truncates toward zero, `rem` takes the sign of the dividend, `<<` is logical-left, `>>` arithmetic-(sign-extending)-right, `>>>` logical-(zero-fill)-right** — and, in **allow** mode, **wrap** silently; the surface `+`/`-` are relational union/difference (no int↔Int coercion), so arithmetic is reached only through the `fun/…` forms. `div[x,0]` (allow mode) is now a fully closed form: **`div[x,0] = -sign(x)`** — `1` for every negative `x` including MIN, `0` at `x=0`, `−1` for every positive `x`; separately, `div[MIN,−1] = -8` (ordinary two's-complement division-overflow wraparound, correcting an earlier probe-spelling artifact that had this at `1`). **Shift-amount masking (pinned mt-044 round 3, §10.7d):** the shift amount for `<<`/`>>`/`>>>` is masked to its low `⌈log2 w⌉` bits before shifting (source: `TwosComplementInt.shl`/`shr`, `32 - Integer.numberOfLeadingZeros(w-1)`, confirmed at bw3/4/5/6 with no discrepancy — a closed bit-length identity, not width-specific); `>>`/`>>>` **never self-overflow** on any operand (own overflow bit hardcoded `FALSE` in source, only pre-existing operand overflow propagates); `<<` sets a genuine overflow flag when the shift truly changes the represented value, but has a **jar-specific defect**: a masked-away ("junk") high bit of the shift amount can spuriously set the overflow flag with zero effect on the actual shifted value, depending on the shiftee's own bit pattern at the position the jar's incremental barrel-shifter circuit happens to inspect for that bit — mettle must replicate the jar's exact per-bit circuit (not just "did the masked value change") to stay bug-compatible; full per-op conditions and decisive probes in §10.7d. In **forbid** mode (mettle's LEDGER-001 default) an overflowing subterm does **not** flatly conjoin `¬overflow`; at the comparison where an `Int` becomes a `Formula` it is guarded by the **Milicevic/Jackson polarity rule** (`DefCond.ensureDef`): at positive polarity an *existentially*-dependent overflow forces the atom **false** (`∧ ¬of`, excluding an overflowing witness) while a *universally*-dependent overflow forces it **true** (`∨ of`, so a `∀` is not falsified by an overflowing binding); negative polarity swaps the two. **Caveat, pinned mt-044/§10.7c–§10.7d (2026-07-19, three follow-up rounds same day — gap-closing, then a shift-semantics + `negate[8]`-confound audit that retracted half of the second round's own finding):** the guard reliably fires for single-quantifier formulas over `Int`/`seq/Int`, and is **conclusively independent of Kodkod's `skolemDepth`** (swept 0 through 4 and off — identical at every value, refuting the original skolemization hypothesis). There is **one** defect, not two: `DefCond.isInt` recognizes *only* the literal bare `Int`/`seq/Int` builtin as "an Int domain" — any quantified variable whose **own** domain is anything else (a comprehension, or a **sig**) has its classification silently default to existential, **unconditionally, regardless of nesting depth (0, 1, 2, or more) or nesting shape (lone `∀`, same-type `∀∀`/`∃∃`, or mixed-type `∀∃`/`∃∀`)** — invisible when the variable is genuinely existential, wrongly excluding-instead-of-rescuing when it is genuinely universal. **This is a live, practically significant wrong-verdict risk**: `all p: SomeSig | <cardinality-overflow comparison over p>` — the single most common corpus shape this entry touches — is confirmed (decisive jar probe: forced-nonempty-field cardinality under a bare `all p:P`, `plus[#p.f,7]>=0`) to get **forbid = UNSAT (excluded)**, not the naive rule's SAT (rescued), with the identical arithmetic shape driven by a bare-`Int` `∀` confirmed to rescue correctly as an isolating control. **A second, independent "nesting-position" defect was reported in the gap-closing round and is RETRACTED here**: every one of its decisive probes turned out to spell an enclosing universal's domain lower bound as `negate[8]` (`=minus[0,8]`, itself an overflowing computation carrying a permanent forbid-mode overflow flag — confirmed: `negate[8]=negate[8]` is forbid-UNSAT, `(0-8)=(0-8)`, the clean peephole-folded MIN literal, is forbid-SAT), which silently forces any `x>=negate[8]` domain guard false for every candidate, **emptying the whole domain** and producing vacuous-truth verdicts (`all n:{empty}|BODY` trivially SAT, `some n:{empty}|BODY` trivially UNSAT) that mimicked a nesting-position effect. Re-probed with the clean `(0-8)` bound, every one of those cells — including the same-type `∀∀` nesting, previously (and wrongly) reported as evidence that same-type nesting is inherently robust — retrodicts as a plain instance of the single defect above, with no separate mechanism needed. One narrow sub-corner is NOT closed by this investigation: an int-ITE or `implies`-antecedent wrapper around the domain-recognition defect's triggering shape sometimes avoids the defect for reasons not traced to source (P9, P12) — see §10.7c's Open note. Full mechanism, retrodiction over every P1–P14 probe (100% match after correction), the decisive probes, the retraction's audit table, and the conservative typed-defer boundary: [§10.7c](docs/reference/alloy6-translation.md#107c-mt-044-mixed-nesting-forbid-mode-trigger-pinned-jar-verified-2026-07-19) and [§10.7d](docs/reference/alloy6-translation.md#107d-mt-044-round-3-shift-semantics-pinned-negate8-forbid-confound-audit-jar-verified-2026-07-19).
**Status:** `approved` (tech lead, no-fork delegation, 2026-07-19; `verified` mt-044 same day — zero open residuals: every named soft spot below was probed shut or amended the Rule, and the one genuinely-jar-unpinned sub-corner (the ITE/`implies` sliver) is carved out of the entry's scope as a standing typed defer per project policy, not a "re-derive later" caveat — mettle never emits a verdict there). No fork: pure match-the-jar, including its pinned defects (`DefCond.isInt` domain-recognition, `<<` junk-bit overflow). Verification history: `proposed` at mt-043 (2026-07-18) with two named residuals; both closed plus one retraction cycle at mt-044 (2026-07-19), as follows. **both previously-named residuals are now closed (mt-044, 2026-07-19, [§10.7b](docs/reference/alloy6-translation.md#107b-mt-044-ledger-005-residual-closure-probes-jar-verified-2026-07-19))**: **(1)** the allow-mode `div[x,0]` value for zero/negative dividends is now a jar-verified closed form (`-sign(x)`, uniform including at MIN — full 16×16 `div`/`rem` sweep confirms Java-exact semantics everywhere `b≠0` except the single `div(MIN,−1)` case, now corrected to `-8`); **(2)** the forbid-mode polarity rule was probed across 15 mixed-polarity models (∀∃/∃∀ nesting, negation/implies/check, double negation, `sum`/int-ITE under a quantifier, mixed-classification atoms) plus 18 follow-up diagnostic probes — 12/15 matched prediction, and the 3 mismatches (plus the follow-ups) surfaced a genuine, reproducible, previously-undocumented gap (the guard not firing for some mixed-type doubly-nested quantifiers, folded into the Rule text above) rather than merely "more coverage needed." ### LEDGER-006 — cardinality `#`
**Rule:** `#e` is a two's-complement `IntExpression` (`cset(e).count()`) at the command bitwidth; it participates in overflow **exactly like arithmetic** — a count exceeding `2^{w-1}−1` **wraps** in allow mode (`#A=8` at bw 4 reads as `−8`) and is an **overflow** in forbid mode (an over-large count excludes the instance under the §11.3 polarity rule). `#` is the only int-producing coercion on a set; there is no implicit set→int elsewhere (no int↔Int coercion — resolution §4.5).
**Status:** `approved` (tech lead, no-fork delegation, 2026-07-18; `verified` mt-043 same day) — zero open residuals; the shared forbid-mode polarity machinery is LEDGER-005's (this entry only pins that `#` feeds it, which is decisively probed at positive polarity). No fork: pure match-the-jar.

### LEDGER-007 — String
**Rule:** The `String` sig's atoms are the set of **string literals referenced by the command** (its formula and every parent command's, plus every reachable sig's appended facts and field-declaration exprs, plus **all top-level module facts**, recursing into **called** funcs/preds only), **padded** — when a `for … but N String` scope is given (which **must be exact**; a non-exact `String` scope is a pre-solve error) — with synthetic atoms named **`"String0"`, `"String1"`, … (NOT `unused%d`)** until the population reaches `N`; if the referenced literals already **exceed** `N` the scope **expands** to hold all of them (so an `exactly N String` scope is effectively `max(N, #referenced)`, not truly exact). With no `String` scope (`maxstring = −1`) the atoms are exactly the referenced literals (no padding). String atoms are ordinary uninterpreted atoms appended **last** in the universe (after sigs and ascending ints); the only operations are `=`/`!=`/`in`/`#` — there is **no richer String algebra in 6.2.0**.
**Status:** `approved` (tech lead, no-fork delegation, 2026-07-18; `verified` mt-043 same day) — zero open residuals. No fork: pure match-the-jar, with one flagged micro-choice — the jar collects into a `HashSet`, so its atom *order* among referenced literals is nondeterministic; mettle chooses its own deterministic order (string atoms are symmetric, so verdict and SB-0 count are provably order-independent).
**Evidence:** [reference/alloy6-translation.md](docs/reference/alloy6-translation.md) §13; probes §10.9 S1–S7, jar-verified 2026-07-18. Source: `Command.getAllStringConstants`, `ScopeComputer.compute`, `A4Solution` ctor (commit `794226dd`). **Discrepancy resolved:** the `unused%d` naming in LIMITATIONS/STATE is wrong — that is the *instance-read-back* label for sig-unclaimed atoms (`A4Solution.rename`), a different mechanism; the padding atoms are `"String" + i` (probe S2). Un-defers the `scope` family (mt-045, `fm2cfs.als`).
**Test:** `crates/als-core/tests/string_conformance.rs` (mt-045, jar-free, CI-safe) pins probes S1–S7 and the solve-level behavior: S1 non-exact-scope reject (`StringScopeNotExact`); S2 `"String0"`/`"String1"`/`"String2"` padding after the int atoms with `String` bound exactly to them; S3 literal + padding; S4 `max(N,#referenced)` expansion; S5 no-scope = exactly referenced; S6 top-level-fact collection; S7 uncalled-pred non-collection vs called-pred collection; plus `some s: String | s = "hello"` SAT, distinct-literal singletons (`"a" != "b"` SAT, `"a" = "b"` UNSAT), and `#String` cardinality against an exact scope. The universe golden `string_scope_pads_after_ints` and the reject `nonexact_string_scope_rejects` also live in `tests/scope.rs`; `tests/lower.rs::string_literal_lowers` pins that a literal now lowers.

### LEDGER-008 — `seq` semantics
**Rule:** `seq/Int` is bound exactly to `{0 … maxseq−1}`, where `maxseq` defaults to `min(overall, 2^{w-1}−1)` (or 4 when no overall) and is set directly, independent of the overall scope, by a `for N seq` clause. A field `f: seq X` desugars to **`f: seq/Int -> lone X`** (stored `owner -> index -> X`, `lone` on the value column) **plus exactly one implicit contiguity fact** — `dom(f) − dom(f).(Int/next) ⊆ Int/zero` — forcing the used indices to be a contiguous prefix from 0; `util/sequniv`/`util/seqrel` functions are otherwise ordinary library funcs over the `seq/Int` domain (only `seq/Int` and the contiguity fact are builtin-special).
**Status:** `approved` (tech lead, no-fork delegation, 2026-07-18; `verified` mt-043 same day) for the builtin-special seq behavior (desugar, contiguity, `maxseq`, `seq/Int` bound) — zero open residuals; no fork. ### LEDGER-009 — `run`/`check` target resolution (own-module-first)
**Rule:** A **bare** (unqualified) `run`/`check` command target `<name>` is resolved **own-module-first**: the jar first searches the command's own (root) module's own preds/funcs (for `run`) or assertions (for `check`) via `getRawQS`, and **only if the own module declares no such name** falls back to `getRawNQS` over **all reachable opened modules**; therefore a same-named pred/fun in the user's own module **shadows** any opened-module candidate, including the auto-opened `util/integer` `add`/`sub`. If the own module has the name it is used unconditionally (never an ambiguity against opened candidates); if the fallback finds **≥2** candidates across opened modules it is `ErrorSyntax "The name \"<name>\" is ambiguous."`; a **qualified** target (`m/name`) uses `getRawQS` only (no NQS fallback).
**Status:** `approved` (tech lead, no-fork delegation, 2026-07-18; `verified` mt-040 same day) — zero open residuals; every corner (own vs auto-opened util, own vs opened user module, own fun vs opened fun, no-own NQS fallback, multi-opened ambiguity, qualified target) probed shut below. No fork: pure match-the-jar. **Implementation now unblocked**: flip the `#[ignore]`d test and add own-module-first to `lookup_run_target` (scheduled into the next convenient lowering-adjacent chunk).
**Evidence:** jar-verified 2026-07-18 (OpenJDK 21, `oracle/org.alloytools.alloy.dist.jar` 6.2.0, sat4j, symmetry 0, noOverflow true; harness `ProbeR4.java`, session scratchpad). Decisive probe pairs (each contrives the two candidates to give **different verdicts**):
- **own pred shadows auto-opened util fun** — `sig S{} one sig A{} pred add{no A} run add for 3` → **UNSAT** (the own pred `no A`, always false, is run; not the `util/integer` `add` fun).
- **own pred, explicit `open util/integer`** — `open util/integer … pred add{some S} run add for 3` → **SAT** count 5 (own pred).
- **own fun shadows opened util fun (no ambiguity)** — `open util/integer … fun add:S{S} run add for 3` → **SAT** count 3 (own fun; no "ambiguous").
- **own pred shadows opened user-module pred** — root `pred dup{no A}` (`one sig A`) over `open usermod` where `usermod` has `pred dup{some T}` → **UNSAT** (root pred).
- **no own candidate → NQS fallback** — `open util/integer … run add for 3` (no own `add`) → **SAT** count 3 (the util fun is used).
- **multi-opened → ambiguous** — `open modA` + `open modB` each with `pred foo`, no own `foo`, `run foo` → `ErrorSyntax "The name \"foo\" is ambiguous."`.
- **qualified opened target fails** — `open util/integer … run util/integer/add for 3` → `ErrorSyntax "The predicate/function \"util/integer/add\" cannot be found."`.
Source: `CompModule.resolveCommand`/`resolveCommands` (`getRawQS(4,·)` then `getRawNQS(this,4,·)` for run; kind `2` for check), commit `794226dd`. mettle's current `lookup_run_target` collects candidates from **every** reachable module with no own-module priority, so an own-module pred colliding with the auto-opened `util/integer` `add`/`sub` yields ≥2 candidates and defers typed (`overloaded run target`, 2 corpus commands: `addressBook.als[0]`, `addressBook1e.als[0]`) — a verdict-neutral defer, never a wrong answer.
**Test:** `crates/als-core/tests/lower.rs::run_target_own_module_first` — jar-pinned, **live** (un-`#[ignore]`d at mt-044, when `lookup_run_target`/`find_assert` gained the own-module-first rule; own-module `pred add` shadowing the auto-opened `util/integer/add` lowers to the user pred).

### LEDGER-010 — integer-equality typing (one-sided `Int[·]` cast) & cast overflow semantics
**Rule:** A relational `=`/`!=`/`in` where exactly one translated side is an `Int[·]` cast is **always Kodkod set equality/subset** — never int promotion (the other side is never summed); bare `+` between int-typed operands stays relational union (the peterson `# priority = # pid + 1` shape is the set-collapse pitfall, forcing `#pid=#priority=1`, and the jar's UNSATs there are bug-compatible set semantics, not arithmetic). Both-sides-cast remains the sole int-compare case (LEDGER-005/§2.2, unchanged). In **forbid** mode, four pinned layers: **(1)** a cast of an overflow-capable operand denotes the **empty set** when the operand overflowed (the jar's cast cells are built by env-less `Int.eq` → per-cell `∧¬of`; polarity- and context-independent — comparisons, unions, `some`/MultTest); **(2)** the comparison additionally applies LEDGER-005's rules-0–3 per-variable classification (bare-Int-∀ dependence **rescues** — first jar-confirmed rescue on the set-eq path — everything else, **including closed/variable-free operands**, excludes) to every capable cast reachable through the compared sides' set-operator structure, unions included; **(3)** the **constant escape**: a cast whose operand is translation-constant (variable-free over exactly-bound relations) contributes **no** comparison-level guard — its constant-empty matrix sheds its DefCond in the jar's matrix fast paths — while the int-compare path *keeps* constant flags; mettle implements this as the shared structural `translation_constant` predicate (encoder = evaluator by construction); **(4)** LEDGER-005's carved-out **ITE/`implies` sliver is closed**: behind an int-ITE (either branch) or in an `implies` **antecedent** (nested antecedents included), a non-bare-Int effective-∀ driver behaves as *correctly classified* (rescue); consequents, `and`-wrapping, and bare negation get ordinary Defect-A treatment. The former blanket typed defer (`eq_typing_defer`, 10 corpus commands) and the rule-4 defer are both retired.
**Status:** `approved` (tech lead, no-fork delegation, 2026-07-21; `verified` same day). No fork: pure match-the-jar, including two further jar defects replicated bug-compatibly (the constant escape; the antecedent/ITE escape). One intermediate rule from the first probe pass ("no free-variable dependency ⇒ no guard") was **refuted and withdrawn** during tech-lead verification — its three supporting cells were all fully-constant translations (layer 3 explains them).
**Evidence:** [reference/alloy6-translation.md](docs/reference/alloy6-translation.md) **§10.7e** (FACTs 1–5, ~52 decisive cells over two rounds, predictions recorded before every run; jar-verified 2026-07-21, OpenJDK 21, 6.2.0 jar, sat4j, sym 0, both overflow settings). Source at commit `794226dd`: `TranslateAlloyToKodkod` `EQUALS`/`IN`/`PLUS`/`MINUS`-peephole (lines 1230–1327), `Int.eq(other) = eq(other, Environment.empty())` (`Int.java:129`), `BooleanMatrix.eq/subset` + `DefCond.ensureDef` (bytecode-verified vs fetched source via `javap`). Probe harnesses `scratchpad/probe/ProbeEqTyping*.java` + `mt051_report.md` (with the correcting review addendum). Out-of-scope corners (documented in §10.7e, LIMITATIONS): casts nested inside `Card`/`sum` operands, casts in quantifier-decl bounds (layer 1 applies; the decl-level guard analog doesn't), `&`/`-` nesting (source-read, not probe-confirmed).
**Test:** `crates/als-core/tests/eq_typing_conformance.rs` (jar-pinned rows for FACTs 1–5 incl. the constant-escape trio R-cardun/T5/T6, the closed-operand cells D4/T1/T4/D6/D3a, the rescue cell, the MultTest cell T7, and the full Part C ITE/implies family); `int_conformance.rs` keeps I9–I13/GAP2 green and its former defer row asserts the rule-derived verdict.
